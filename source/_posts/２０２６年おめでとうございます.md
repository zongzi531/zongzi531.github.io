## 低代码迭代

### 自提供 Agent 分析能力

之前受业务牵连，Chat BI Agent 由业务后端实现，会导致逐渐定制化，并且无法横向发展。

考虑到这个背景，转为兼容升级自提供 Agent 能力，支持生成仪表盘、单个图表，对图表进行分析能力，同时暴露 MCP 生成仪表盘功能。

以加强低代码在仪表盘和报表（后续规划）场景领域的 AI 落地能力。

### 网格布局计算优化

在实际的使用过程中发现每个网格 `LayoutItem` 的 Gap 距离计算存在小概率交互差异问题，导致用户在使用过程中产生意外的困惑。

重新分析计算逻辑进行如下优化（前提条件：一行有 12 栅格）：

现状是，假设一行目前存在 2 个 `LayoutItem` ，并且设每个 `LayoutItem` 的宽度 `w` 为 `x` ，设一整行为 `width` 。

那么计算公式则得出： `width = 2x + 3Gap` 。看上去没有问题吧，我一开始也觉得没问题，但是当仔细设计会发现其实这个公式中的 `x` 的值包含了不属于 `LayoutItem` 的 Gap 距离在内，导致交互产生意外。

我们重新来看待这个问题，其实一行的宽度计算公式应该是如此，设每个 `LayoutItem` 的宽度为 `i` ，则整行宽度为 `width = 12i + 13Gap` 。

还是上面这个假设，假设一行存在 2 个 `LayoutItem` ，则最后得出的公式应该是 `width = 12i + 13Gap` ，但是在表述其中一个 `LayoutItem` 的宽度时。

公式应该是： `x = 6i + 5Gap` 。所以之前的问题就是 `x` 中包含了不需要参与计算的 5 个 Gap 距离。从而导致交互时出现意外结果。
